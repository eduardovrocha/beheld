## ⚠️ Concerns (5 med, 4 low — zero high)

### MED — `bundle.rekor.uuid` does not match Rekor's entryUUID for direct lookup

`bundle.rekor.uuid` stores `sha256(canonicalizedBody)`, but Rekor's actual
entryUUID for direct API lookups is different. The real entryUUID for
logIndex 1704801591 is
`108e9186e8c5677ae9e4766d0dae92a2e0cfcc0188e1be13a002ca5eca1322c3521e14a928248913`.
A verifier following `bundle.rekor.uuid` alone would fail. Lookup by
logIndex still works.

**Source:** signature chain scout, evidence 12 + concern 1

### MED — `GET /v/{slug}` returns HTML by default

Only yields JSON when `Accept: application/json` is sent. No stable
machine-readable path like `/v/{slug}.json` or `/api/v/{slug}` exists.
External verifiers using plain `curl` get HTML and must know about the
Accept-header content negotiation.

**Source:** prod JSON scout concern 2 + signature chain scout evidence 1

### MED — `payload.enrichment.ecosystems = {}` (empty)

`payload.enrichment.ecosystems` is an empty object while
`payload.core.ecosystems` has 4 entries. Possible contract gap between
enrichment and core layers — enrichment should reflect harness-observed
ecosystem usage independently of git-history presence.

**Source:** prod JSON scout concern 3

### MED — Prompt-length signals not reaching enrichment

`payload.enrichment.workflow_metrics.prompt_avg_chars=0` and
`prompt_median_chars=0` despite 50 sessions and `prompt_quality=37`.
The prompt-length data clearly exists upstream (PromptQuality scored 37,
which is non-trivial), but is not surfacing in `workflow_metrics`.

**Source:** prod JSON scout concern 4

### LOW — Two Ed25519 signatures in the trust chain

`bundle.signature` (raw over canonical JSON) AND the DSSE-PAE signature
inside the Rekor entry. Both bind to the same key but are different bytes.
Verifiers using cosign / rekor-cli will see one; verifiers using openssl
or pynacl on the canonical payload will see the other. Needs documentation
so users know which flow they're invoking.

**Source:** signature chain scout concern 2

### MED — Dual versioning

`envelope.version=7` coexists with `payload.signals.schema_version=1` and
the page title says "v3 schema". Three different version markers in one
bundle. Confirm intentional or pick one canonical version field.

**Source:** prod JSON scout concern 5

### LOW — `core.total_commits` vs `stack.total_commits_analyzed` asymmetry

`core.total_commits=1563` vs `stack.total_commits_analyzed=1222`. The SPA
only renders the stack figure, hiding the larger core-scope number from
users. Not a render drift but a hidden bundle-internal asymmetry.

**Source:** render parity scout

### LOW — `PLATFORM_LABEL` missing entries

`PLATFORM_LABEL` at `snapshot-html.ts:93-99` does not map `testing`,
`database`, `ci_cd`, `mobile`, `blockchain`. These render as raw lowercase
keys alongside properly-cased "Docker" / "GitHub", producing mixed
capitalization in the platforms list.

**Source:** render parity scout

### LOW — `WORKFLOW_LABEL` dead code

Workflow section at `snapshot-html.ts:348-352` renders raw bundle keys
(`debug-driven`, `exploratory`) instead of using `WORKFLOW_LABEL` defined
at line 116+. Suggests dead code or missing wire-up.

**Source:** render parity scout

## Methodology

- **Scout 1 — prod JSON validation.** Fetched the bundle via `curl
  --compressed -H "Accept: application/json"` and asserted shape contracts
  (envelope version, payload.core / .enrichment / .scores presence + type
  checks, harness_sources enum membership, Rekor fields, hash / signature
  / public_key prefix conventions).
- **Scout 2 — signature chain.** Independently verified the Ed25519
  signature via BOTH pynacl (`VerifyKey.verify`) and openssl
  (`pkeyutl -verify -rawin`); independently computed
  `sha256(canonical_json(payload))` and compared to `bundle.hash`;
  resolved the Rekor entry by logIndex and confirmed `integratedTime`,
  `payloadHash`, and verifier public key match the bundle's claims.
- **Scout 3 — render parity.** For each of the 10 user-visible claims
  in the SPA's rendered profile, traced back to the source field in the
  bundle JSON and the renderer line in `snapshot-html.ts`. Confirmed CLI
  and web renderers byte-identical except for one import path
  (`./tier` vs `../lib/tier`).
- **Synthesizer.** Produced verdict + concerns + backlog from the three
  scout reports under a structured schema (`verdict`, `verdict_reason`,
  `what_works`, `concerns[]`, `backlog[]`, `win`).

Full raw output (12 evidence claims for the signature scout alone, byte
counts, hex strings, command-line invocations) preserved at the
workflow's transcript directory.
