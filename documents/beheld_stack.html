Stack técnica sugerida para o Beheld, organizada por camada

Stack sugerida — Beheld

Organizada por camada. Prioriza tecnologias que você já domina e minimiza superfície nova.

camada 1 — coleta e hooks

TypeScript + Node.js MCP server local

O SDK oficial do MCP é TypeScript-first. O server roda como processo daemon local (porta 7337), recebe eventos do Claude Code e do Continue.dev, e grava JSONL em ~/.beheld/sessions/. Você já escreve TS no frontend — mesma linguagem, sem contexto novo.

MVP

VS Code Extension API (TypeScript) extensão VSIX

Extensões VS Code são obrigatoriamente TypeScript/JavaScript. Registra listeners de workspace, window e commands. Mostra ícone na status bar, painel de onboarding e badge de score. Comunica com o MCP server local via HTTP.

MVP

Claude Code hooks (JSON config) hook nativo

Configuração pura em ~/.claude/settings.json. O CLI instala o hook automaticamente no beheld init. Não requer código extra — só apontar para o MCP server local.

MVP

camada 2 — processamento e scoring

Python scoring engine

Você já usa Python para scripts e data extraction. É a melhor linguagem para o pipeline de análise: lê os JSONL, roda o classificador de padrões, calcula os scores e salva no SQLite local. Roda como job periódico disparado pelo MCP server (a cada sessão encerrada).

MVP

SQLite armazenamento local

Zero configuração, zero servidor, arquivo único em ~/.beheld/profile.db. Perfeito para local-first. Quando o usuário ativa o sync, o backend lê este arquivo diretamente ou recebe o delta via API. Suporta até anos de histórico sem problemas de performance.

MVP

Anthropic API (claude-sonnet-4-6) classificação e insights

Chamada pontual ao fim de cada sessão para classificar tipo de projeto e gerar insights. Roda localmente no pipeline Python. Custo marginal por sessão (~0.01 USD). Pode ser substituído pelo Ollama com qwen2.5-coder:14b para quem preferir zero custo variável — você já tem o setup funcionando.

MVP

camada 3 — CLI

TypeScript + oclif ou commander.js CLI global

Publicado como npm i -g beheld. Gerencia init, view, sync, delete. Chama o Python scoring engine via child_process ou via HTTP se o daemon estiver rodando. Mesma linguagem do MCP server — um único repositório.

MVP

camada 4 — plataforma web (v1)

Ruby on Rails 7 API backend

Sua stack principal. Recebe os deltas de sync via API JSON, armazena perfis, calcula benchmarks entre usuários e serve os insights. ActiveRecord + PostgreSQL. Autenticação via magic link — sem senha, sem fricção. Você já tem padrões de JWT e autenticação bem estabelecidos.

v1

React 18 + Zustand dashboard web

Seu frontend principal. Dashboard de perfil, timeline de evolução, benchmarks. Zustand para estado global (você já usa). Recharts para os gráficos de evolução. Mesma base do seu projeto de biblioteca — pode reaproveitar estrutura, i18n e padrões de teste com Playwright.

v1

PostgreSQL + Redis banco + cache

PostgreSQL para perfis, sessões e benchmarks. Redis para cache de rankings e fila de processamento de sync (Sidekiq). Stack que você já opera — sem surpresa em produção.

v1

decisões e trade-offs

Python vs TypeScript no scoring
+ Python tem ecossistema superior para análise de texto e ML
- Dois runtimes no MVP (Node + Python). Mitigação: scoring roda como microserviço HTTP separado, CLI chama via porta local.
SQLite local vs PostgreSQL
+ SQLite: zero config, perfeito para local-first, portátil
- Sync requer serialização do delta. Mitigação: exporta JSON incremental assinado, backend valida e importa.
Anthropic API vs Ollama
+ API: qualidade superior, sem setup do usuário
- Custo variável e dependência externa. Mitigação: Ollama como fallback configurável — você já tem qwen2.5-coder:14b rodando.
Rails vs Node no backend
+ Rails: sua stack dominante, produtividade máxima, padrões já estabelecidos
- Poderia ser tudo TypeScript/Node. Mas Rails entrega o backend v1 em metade do tempo para você especificamente.
