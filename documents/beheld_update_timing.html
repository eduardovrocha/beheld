Estratégias de timing para atualização do perfil Beheld

Quando atualizar o perfil?

Três momentos possíveis — cada um serve um propósito diferente.

os três momentos e seus trade-offs

1

Início da sessão

hook PreToolUse

O agente lê o perfil atual antes de começar a trabalhar. Útil para contextualizar a sessão — ex: "você tem TDD baixo, quer praticar hoje?". Mas o score exibido é sempre do dia anterior — a sessão que está começando ainda não foi analisada.

+ Contexto disponível antes de qualquer trabalho

+ Pode influenciar o comportamento da sessão atual

- Score é stale — não reflete a sessão que está iniciando

- Adiciona latência antes de qualquer interação

2

Final da sessão

hook Stop

O scoring engine processa os eventos da sessão que acabou de encerrar e recalcula todos os scores. O perfil sempre reflete o trabalho mais recente. É o momento mais natural — a sessão acabou, o dado está completo.

+ Score sempre atualizado com dados completos da sessão

+ Não interfere no fluxo de trabalho

+ Dado completo — sessão encerrada é analisada inteiramente

- O usuário não vê o impacto da sessão atual enquanto trabalha

3

Contínuo (durante a sessão)

polling periódico

Engine reprocessa os eventos a cada N minutos enquanto a sessão está ativa. O score vai subindo em tempo real conforme o usuário trabalha. Efeito motivacional interessante, mas complexidade técnica maior e risco de distração ("gamificação indesejada").

+ Feedback imediato — motiva boas práticas na hora

+ Usuário vê impacto de cada decisão (escrever teste → score sobe)

- Sessão incompleta = dado parcial = score impreciso

- Risco de distrair do trabalho real

- Maior consumo de CPU (reprocessamento frequente)

estratégia recomendada — combinação dos 3 momentos

Linha do tempo de uma sessão típica

Sessão inicia (PreToolUse)

Lê perfil do SQLite — sem chamar a IA. Exibe score anterior em <50ms.

Durante a sessão

Coleta silenciosa. Nenhum reprocessamento. Nenhuma notificação.

Sessão encerra (Stop hook)

Engine processa sessão completa. Recalcula os 4 scores. Grava no SQLite.

Notificação (1x por dia)

Se for a primeira sessão encerrada no dia: notificação nativa com delta de score.

Próxima sessão

Lê o score atualizado. Ciclo reinicia.

O que acontece em cada momento

Início — leitura rápida (sem IA)

SELECT do SQLite. Retorna scores salvos do último processamento. Latência <50ms. Disponível para o /beheld e para a sidebar do Continue.dev.

Final — processamento completo

Engine lê os novos eventos JSONL, extrai padrões, recalcula scores, atualiza SQLite. Chamada à IA apenas se insights estiverem expirados (>24h).

Insights — gerado assincronamente

Não bloqueia o processamento de score. Roda em background após o Stop. Cache de 24h — não chama a IA a cada sessão.

Update de versão — verificado 1x/dia

Junto com o processamento do Stop, verifica se há nova versão. Se sim, avisa na notificação. Nunca bloqueia nem atualiza sozinho.

leitura vs. cálculo — distinção importante

Quando o usuário digita /beheld

Leitura instantânea do SQLite.
Nunca dispara processamento.
Nunca chama a IA.
Retorna o estado mais recente já calculado.

leitura pura

Quando a sessão encerra (Stop)

Processamento completo em background.
Scores recalculados e gravados.
Insights regenerados se cache expirado.
Notificação enviada se primeira do dia.

cálculo completo

Quando o usuário executa beheld view

Leitura do SQLite + opcionalmente dispara processamento se há eventos não processados desde o último Stop. Flag --refresh força reprocessamento manual.

leitura + refresh opcional
